# Glossary

PIC uses theory-specific terms, but each term has a practical software meaning.

| PIC term | Common term |
| --- | --- |
| capability packet | checked reusable work item |
| verified packet capital | reusable work item accepted by verifier, identity, rollback, and residual policy |
| residual ledger | explicit unresolved-work ledger |
| external proof obligation | evidence request that PIC cannot settle without a domain verifier |
| SQOT | finite attention and task scheduler |
| ALT | reusable abstraction value checker |
| ECPT | workflow graph and collective phase-control model |
| BIT | bottleneck and witness calculus |
| TRC | typed trace and real-world frontier compiler |
| agent check compact mode | short CI/agent JSON for first-pass output checking |
| agent runbook | deterministic list of next commands, schemas, and fields to inspect |
| phase control audit | typed ECPT report for proxy target, baseline, and queue/capacity checks |
| frontier debt report | typed TRC report for unresolved frontier and trace debt |
| bottleneck witness report | typed BIT report for a reusable bottleneck-release task |
| value bridge report | typed ALT report separating proxy metrics from checked downstream value evidence |
| agent message delivery report | local peer-message send/receive/verify result with nonce and residual checks |
| agent relay readiness report | local inbox communication readiness report |
| live connector default mode | explicit-source bounded candidate intake |
| theory fidelity report | canonical-suite summary of coverage, snapshot health, and finite-upgrade candidates |
| Psi dashboard | finite workflow health vector |
| collective phase certificate | protocol-relative workflow certificate over a fixed agent population |
| ASI-proxy phase-control | bounded proxy workflow for organizing capability evidence, not a proof of real ASI |

Operational rule: `accepted=true` means the checker accepted the scoped JSON and
finite checks. `workflow_usable=true` means the output is useful for routing or
onboarding. `operationally_usable=true` is stricter and profile-dependent.
`settled=true` is reserved for scoped obligations actually discharged by
verifier rules.

Live-capable defaults mean explicit sources can be fetched or relayed under
byte/page/policy limits. They do not grant background crawling, shell execution,
repository mutation, or promotion from candidate packet volume to verified
capital.
